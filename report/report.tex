\begin{abstract}
This report presents the design and implementation of a wireless keyboard system that transmits keystrokes from a PC to a 16$\times$2 LCD display driven by an STC89C52RC microcontroller. The STC89C52RC, based on the legacy 8051 architecture, lacks hardware SPI and has limited GPIO availability due to port multi-connection conflicts on its breakout board. To work around these constraints, an ESP32 microcontroller is introduced as a serial-to-radio bridge, forwarding characters received over USB serial to the STC89C52RC via a pair of NRF24L01 2.4\,GHz radio modules. The report details the development toolchain setup, the incremental experiments used to bring up and debug the radio link, and the workarounds required for bit-banged SPI initialization timing and clone NRF24L01 module reliability. The final system supports real-time character display with 5$\times$ packet redundancy and sequence-number deduplication.
\end{abstract}

\section{Introduction}

The STC89C52RC is an 8-bit microcontroller based on the Intel MCS-51 (8051) architecture, a design dating back to the early 1980s. While it remains widely used in embedded systems education and low-cost applications, its age brings significant hardware constraints that complicate modern peripheral integration. A central challenge arises not from the chip itself, but from its \textbf{breakout board circuit}. While the STC89C52RC exposes four 8-bit I/O ports (P0 to P3), the development board routes several of these pins to on-board peripherals, creating \textbf{port multi-connection} conflicts that leave few GPIO lines available for additional devices. Beyond port scarcity, the STC89C52RC lacks hardware SPI, meaning all communication with the NRF24L01 must be bit-banged in software, a process that is both slower and more sensitive to timing than a dedicated peripheral would be. The chip also has no native USB support; serial communication with a host PC relies on an external CH340G USB-to-UART converter. These limitations make it impractical for the STC89C52RC to directly interface with modern peripherals such as USB keyboards, Wi-Fi modules, or Bluetooth radios.

The goal of this project is to build a \textbf{wireless keyboard} system: the user types on a PC, and the typed characters appear in real time on a 16$\times$2 LCD display connected to the STC89C52RC, with the data transmitted wirelessly over a 2.4\,GHz radio link. This demonstrates how a resource-constrained 8051 microcontroller can be extended with wireless capabilities despite its architectural limitations, and provides a practical, visual application of embedded radio communication.

To avoid further aggravating the breakout board's port conflicts by also wiring a direct serial connection to the PC, the plan is to introduce a second microcontroller, the \textbf{ESP32-DevKitC~V4}, as a dedicated wireless bridge. A pair of NRF24L01 2.4\,GHz radio modules will handle the wireless link between the two boards. The ESP32, acting as a UART peripheral to the PC, will receive keystrokes over serial and forward them as radio packets. The STC89C52RC will receive these packets and display the characters on the LCD. This three-tier architecture offloads the complex serial-to-radio bridging to the ESP32, which has ample resources for the task, while the STC89C52RC focuses solely on receiving packets and driving the display.

\section{Setup}

The development environment required toolchains for two different architectures (the ESP32 and the STC89C52RC) as well as a flashing workflow for each.

For the ESP32, the project uses PlatformIO with the Arduino framework. PlatformIO manages the Espressif~32 toolchain and the RF24 library dependency (v1.4.10) automatically. Building and flashing is a single command: \texttt{pio run -t upload -{}-upload-port COMxx}. The active firmware is selected by copying the desired source file into \texttt{src/main.cpp} before building, since PlatformIO compiles all \texttt{.cpp} files in the \texttt{src/} directory.

For the STC89C52RC, the Keil $\mu$Vision~5 C51 toolchain is used. Obtaining a working installation required a non-commercial license from \texttt{mdk-preview.keil.arm.com}, as the unlicensed version limits the producible build size. Keil can be driven from the command line with \texttt{UV4.exe -j0 -r STC89C52.uvproj} for headless rebuilds. Once compiled, the resulting hex file is flashed to the board using \texttt{stcgal}, an open-source ISP programmer for STC microcontrollers. The STC-ISP protocol requires a power-cycle to enter the bootloader; \texttt{stcgal} automates this by toggling the CH340G serial adapter's DTR pin. A custom patch script (\texttt{patch\_stcgal.py}) inverts the DTR assertion polarity to match this board's circuit. With the patch applied, flashing proceeds with \texttt{stcgal -P stc89 -p COMxx -a STC89C52.hex}, followed by a manual power cycle so the NRF24L01 gets a clean reset.

The complete build-and-run pipeline for the wireless keyboard is: (1)~build and flash the ESP32 with the keyboard forwarder firmware via PlatformIO; (2)~build the STC89C52RC firmware in Keil and flash it via \texttt{stcgal}; (3)~power-cycle the STC board; (4)~run the Python keyboard client, which connects to the ESP32's serial port and begins forwarding keystrokes.

\section{Experiments}

The system was developed and validated through a series of incremental experiments, each building on the results of the previous one.

The first step was to verify that the ESP32 could reliably receive characters from the PC over its USB serial link. A simple Python script using \texttt{msvcrt.getch()} captured individual keystrokes on Windows and forwarded them to the ESP32 at 115\,200\,baud. The ESP32 firmware echoed received bytes back to the serial monitor, confirming that the serial path worked end to end before any radio hardware was involved.

With serial confirmed, two ESP32 boards were each connected to an NRF24L01 breakout circuit (Figure~\ref{fig:esp32-nrf}) to test radio communication in isolation. The transmitter (\texttt{tx.cpp}) sent a 32-byte packet every second containing a 16-bit counter and a \texttt{0xAA}/\texttt{0x55} marker pattern. The receiver (\texttt{rx.cpp}) printed incoming packets as a hex dump over serial and toggled an LED on each reception. Both boards used the RF24 Arduino library over hardware SPI (VSPI: CE on GPIO\,4, CSN on GPIO\,5, SCK on GPIO\,18, MOSI on GPIO\,23, MISO on GPIO\,19). Radio parameters were set to channel~108, 1\,Mbps data rate, 2-byte CRC, no auto-acknowledgement, and a fixed 32-byte payload. Communication was confirmed in both directions by swapping TX/RX roles.

The next step was to replace one of the ESP32 receivers with the STC89C52RC board (Figure~\ref{fig:stc-board}). The NRF24L01 chip was inserted into the STC breakout board and wired via bit-banged SPI on Port~1 (CE on P1.2, CSN on P1.3, SCK on P1.7, MOSI on P1.1, MISO on P1.6, with 470\,$\Omega$ series resistors). The ESP32 continued transmitting its counter packets. Initial attempts showed no indication of reception on the STC side. Online suggestions pointed to a missing decoupling capacitor on the NRF24L01 supply; a 47\,$\mu$F capacitor was soldered in with a mediocre setup, but communication still failed. To rule out the ESP32 side, a second ESP32 and breakout board were used to confirm that the radio link worked between two ESP32s. The problem was therefore isolated to the STC board.

\begin{figure}[ht]
\centering
\includegraphics[width=0.55\textwidth, frame]{images/stc board + components.jpeg}
\caption{STC89C52RC breakout board with LCD1602 display and NRF24L01 module connected.}
\label{fig:stc-board}
\end{figure}

An LCD1602 screen was connected to the STC board (8-bit data on P0, control lines on P2) to provide real-time debugging output. The firmware was modified to display NRF24L01 register values on the LCD after initialization. This revealed that certain register writes were being silently dropped. The channel register (\texttt{RF\_CH\,=\,108}) remained at its default value despite being written. Adding a 5\,ms delay before the configuration sequence allowed the first write to succeed. A second dropped register (auto-acknowledgement disable, \texttt{EN\_AA\,=\,0}) was fixed by inserting a dummy \texttt{STATUS} register read before the configuration writes, effectively waking the SPI bus. With both fixes in place, the STC began receiving packets from the ESP32, and the 47\,$\mu$F capacitor was confirmed to be unnecessary. Bidirectional communication was then verified by temporarily converting the STC to a transmitter.

With radio communication working, the pinging ESP32 firmware was replaced with a keyboard forwarder (\texttt{keyboard\_tx.cpp}). Instead of transmitting a counter, the ESP32 now reads characters from its UART, batches them over a 20\,ms window, and packages them into a 32-byte payload: byte~0 holds the character count (1 to 30), bytes~1 to~30 hold the ASCII data, and byte~31 carries a sequence number. Each packet is transmitted 5 times with 1\,500\,$\mu$s spacing to compensate for the unreliability of clone NRF24L01 modules. On the STC side, the receiver firmware was rewritten to interpret incoming characters and render them on the LCD with cursor management, supporting backspace, enter (line break or clear), and Ctrl+L (full clear). A sequence-number-based deduplication mechanism discards repeated copies of the same packet. The Python keyboard client was extended to display ESP32 debug output and buffer rapid keystrokes.

The ESP32 uses its hardware VSPI interface: CE on GPIO\,4, CSN on GPIO\,5, SCK on GPIO\,18, MOSI on GPIO\,23, MISO on GPIO\,19, with VCC supplied at 5\,V through the breakout board's on-board regulator. The STC89C52RC uses bit-banged SPI on Port~1: CE on P1.2, CSN on P1.3, SCK on P1.7, MOSI on P1.1, MISO on P1.6, all through 470\,$\Omega$ series resistors for voltage protection. The LCD1602 occupies Port~0 (data bus) and three pins on Port~2 (EN on P2.7, RS on P2.6, RW on P2.5).

\section{Discussion}

The main challenge was bringing up the NRF24L01 on the STC89C52RC. Unlike the ESP32, which has a mature RF24 library and hardware SPI, the STC relies on a bit-banged SPI implementation that proved sensitive to initialization timing. Register writes issued too early after power-on were silently dropped, producing a radio configuration that looked correct in code but did not match what was actually loaded into the chip. Because the NRF24L01 provides no error feedback on failed writes, the only way to diagnose the problem was to read registers back and display them on the LCD. This trial-and-error debugging cycle consumed the majority of the development effort. The initial suspicion of a missing decoupling capacitor on the NRF24L01 supply turned out to be a red herring; the root cause was purely a software timing issue in the SPI initialization sequence.

A secondary challenge was the flashing workflow for the STC89C52RC. The STC ISP protocol requires the MCU to be power-cycled to enter its bootloader, and the CH340G serial adapter's DTR pin, which automates this, had inverted polarity on the development board. This required patching the \texttt{stcgal} flashing tool at the source level. Additionally, after flashing, the NRF24L01 remains powered in whatever state it was last configured in, since only the MCU is reset. The initialization code must therefore force the radio into a known state (power-down, flush FIFOs, clear flags) before applying a fresh configuration.

Clone NRF24L01 modules introduced their own reliability issues. The 250\,kbps data rate, which should be supported by the chip, silently fails on clones and defaults back to 1\,Mbps. Auto-acknowledgement, another standard feature, proved unreliable. Both were worked around by fixing the data rate at 1\,Mbps, disabling auto-ack entirely, and instead transmitting each packet five times with sequence-number deduplication on the receiver.

The current system has several limitations that would need to be addressed for further extension. The LCD is limited to 16$\times$2 characters, providing no scrollback or history. Communication is unidirectional (PC to STC only), so the receiver has no way to signal errors or request retransmission back to the sender. The 5$\times$ redundancy scheme increases reliability but also increases channel occupancy and latency; a proper acknowledgement protocol would be more efficient if reliable NRF24L01 modules were used. The keyboard client is Windows-specific due to its use of \texttt{msvcrt} for keystroke capture. Finally, the breakout board's port multi-connection constraints leave very few free GPIO pins, making it difficult to add further peripherals (such as status LEDs or additional sensors) without redesigning the board layout.

\section{Conclusion}

This project demonstrated that a resource-constrained 8051 microcontroller can be extended with wireless communication capabilities using low-cost NRF24L01 radio modules. By introducing an ESP32 as a serial-to-radio bridge, the system avoids the port conflicts that would arise from wiring additional peripherals directly to the STC89C52RC breakout board. The resulting wireless keyboard allows a user to type on a PC and see characters appear on the STC's LCD display in real time over a 2.4\,GHz radio link. The main engineering effort lay in debugging the bit-banged SPI initialization on the STC side, where silent register drops required careful timing workarounds. Clone NRF24L01 module quirks were mitigated through packet redundancy and sequence-number deduplication. Despite the limitations of the current implementation, the system serves as a working proof of concept for wireless peripheral integration with legacy microcontrollers.
